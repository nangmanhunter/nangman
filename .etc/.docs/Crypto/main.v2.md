

##
- 대형브라우저.`http`.❌`window.crypto`-아싸리 막아버린격.


##
네, 결론부터 말씀드리면 **철저하게 브라우저(W3C 웹 표준 기구와 크롬, 사파리 같은 브라우저 개발사들)가 합의해서 막아버린 것**이 맞습니다.

보안 계의 대원칙인 "안전하지 않은 주소(HTTP)에는 강력한 무기를 쥐여주지 않는다"는 규칙 때문인데요, 브라우저들이 왜 이렇게 야박하게 구는지 그 핵심 이유 2가지를 쉽게 정리해 드릴게요.

---

### 1. "중간자 공격(Man-in-the-Middle)" 때문

HTTP는 암호화되지 않은 날것의 데이터가 오가는 통로입니다. 즉, 카페 공용 와이파이 같은 곳에서 나쁜 마음을 먹은 해커가 중간에서 내 패킷을 훔쳐보거나 훔쳐보는 걸 넘어 데이터를 변조(조작)할 수 있습니다.

만약 HTTP에서도 `window.crypto` 같은 강력한 암호화 함수를 쓸 수 있게 허용한다면 어떤 일이 벌어질까요?

1. 사용자는 "와, 이 사이트는 브라우저 내장 암호화 기능을 쓰니까 내 비밀번호가 안전하게 해시(Hash)되겠지?" 하고 안심합니다.
2. 하지만 중간에 침입한 해커가 사용자의 브라우저로 들어가는 자바스크립트 코드를 가로채서 `window.crypto` 대신 **자기가 만든 짝퉁 암호화 함수(비밀번호를 자기 서버로 가로채는 코드)로 바꿔치기**를 해버릴 수 있습니다.
3. 결과적으로 사용자는 안전하다고 착각하지만, 실제로는 보안이 완전히 뚫려버리는 치명적인 결과가 생깁니다.

브라우저 제조사들은 **"보안 기능인 것처럼 속이는 사기극"을 원천 차단**하기 위해, 조작 가능성이 있는 HTTP 환경에서는 아예 이 기능 자체를 제거해 버린 것입니다.

---

### 2. 브라우저들의 "HTTP 죽이기 (HTTPS Everywhere)" 전략

구글 크롬, 애플 사파리, 모질라 파이어폭스 등 현대 브라우저들은 수년 전부터 "웹 표준의 모든 중요 기능은 HTTPS에서만 작동한다"는 전제를 깔고 가고 있습니다. HTTP 웹사이트들을 역사의 뒤안길로 사라지게 만들기 위한 압박 카드인 셈이죠.

실제로 `window.crypto` 뿐만 아니라, 사용자의 민감한 정보를 다루는 아래 기능들도 **HTTP에서는 전부 차단**됩니다.

| 기능 분류 | 기능 이름 | HTTP 작동 여부 | 이유 |
| --- | --- | --- | --- |
| **보안/인증** | Web Crypto API (`window.crypto`) | ❌ **불가** | 암호화 조작 방지 |
| **위치 정보** | Geolocation API (`navigator.geolocation`) | ❌ **불가** | 사용자 위치 탈취 방지 |
| **미디어** | 카메라 및 마이크 접근 (`getUserMedia`) | ❌ **불가** | 도청 및 몰래카메라 방지 |
| **차세대 웹** | 서비스 워커 (Service Worker / PWA) | ❌ **불가** | 브라우저 백그라운드 변조 방지 |

---

### 💡 요약하자면

* **HTTP:** 배송 중에 누군가 박스를 열어서 내용물(코드)을 바꿀 수 있는 택배 시스템
* **HTTPS:** 철통 보안 금고에 넣어서 봉인 인장까지 찍어 보내는 택배 시스템

브라우저는 "누군가 중간에서 코드를 바꿨을지도 모르는 HTTP 주소는 신뢰할 수 없으니, 위험한 기능(암호화, 카메라, 위치 정보 등)은 전부 자물쇠를 잠가버리겠다!"라고 선언한 것입니다. 개발할 때 `localhost`만 예외로 둔 건 "자기 컴퓨터 안에서 개발하는 건 해킹 위험이 없으니 봐줄게~"라는 브라우저의 마지막 배려인 거죠.
